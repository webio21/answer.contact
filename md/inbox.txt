## Message/Email Status
| Status            | Item Background Color | Description
───────────────────────────────────────────────────────── 
| Read              | None         | Read message/email.
| Unread            | Light Yellow | Unread message/email when AI Draft is disabled.
| Draft             | Light Green  | Manually edited draft when AI Draft is disabled.
| Sent              | Light Green  | Manually sent message/email when AI Draft is disabled.
| Draft Ready       | Light Green  | Automatically generated when AI Draft is enabled.
| Auto-sent         | Light Blue   | Automatically generated and automatically sent after the countdown when AI Draft is enabled.
| Pending Auto-send | Light Blue   | Automatically generated reply currently counting down before sending.
| Reviewed and sent | Light Green  | Automatically generated draft that was taken over, edited by the user, and then sent.
| Action Needed     | Pink         | Sending failed or an unexpected error occurred.

- Different statuses are represented using different background colors and status labels.
- For Action needed items, users can reopen the message to regenerate or edit the draft before sending again.

## Expires After Dropdown
   Never

* The system should save the user's Expires After preference when sending an email and automatically select it for future new messages.
* The [Expires After ▾] and [Stop sharing] options are only available when editing a sent Live Mail message.
* Users can modify the expiration period of a sent Live Mail message. The system automatically recalculates the expiration date based on the new setting.
* Changes to the expiration date and stopping sharing take effect immediately without requiring users to click Save.
* When editing a sent Live Mail message, the [Send] button changes to [Save]. The [Save] button remains disabled until the live mail content is modified.

When the user clicks Regenerate to create a new draft version, a new entry is immediately added to the top of the Previous Drafts list in a generating state. During generation, the loading spinner on the left of "Generating new version..." continues rotating.

After the new version has been generated, the Current Draft remains unchanged. In other words, the newly generated version must never replace the Current Draft automatically.

## Draft Version Management

* Clicking [Compare] opens a side-by-side split-screen view comparing the current draft with the selected historical draft version. The left panel displays the current draft in an editable state, with a [Continue Editing] button at the bottom to return to the current draft. The right panel displays the selected historical version in a read-only preview state, with a [Use this version] button at the bottom to replace the current draft content with this version. The previous draft content will be automatically saved as a new historical draft version.
* Clicking a version title opens the full content of that historical draft. Historical drafts can be edited directly and include Undo, Redo, and Copy actions.

The Remote Content Block is a dynamically generated image rendered by the backend from the latest version of the email. Its height automatically adapts to the rendered content so the Live Mail preview can be displayed as completely and consistently as possible across most email clients.

The <img> element is only the visual preview of a Live Mail. The complete, always-up-to-date content exists only at the Live Mail URL.

Uses 32-character Base62 IDs for tracking events and content resources.

* Tracking Pixel
<img src="https://livemail.me/mail/track?id=7Kp3Vn8Qa2Lx9Rm4Tz8Wp6Hs3Nd5YqF2" width="1" height="1" style="display:none;">

* Dynamic Content Image
<img src="https://livemail.me/mail/content?id=7Kp3Vn8Qa2Lx9Rm4Tz8Wp6Hs3Nd5YqF2" style="max-width:100%;height:auto;">

Embedded in the email body, this acts as a controlled email entry point, allowing the sender to modify the content of a sent email even after it has been delivered.
Live Mail is simply a content layer built on top of existing email.

Live Mail provides the following capabilities:
* Track email read/open events.
* Keep the email content under server-side control, allowing it to be updated after it has been sent.
* Stop sharing the email content at any time.

The greatest value of Live Mail is not the preview itself. When recipients open it, they are taken directly into the sender's Contact space, where they can immediately access all available communication channels (Message, Meet, Live Chat, Contact Card, and more), significantly increasing opportunities for ongoing interaction.

Possible backend rendering implementations include:

HTML-to-image rendering
Headless Chromium
Dedicated image rendering services
Canvas / Puppeteer-based rendering pipelines

## Inbox Structure

The Inbox should contain only two sections:
* Timeline
* Trash

Traditional mailbox folders such as Sent, Drafts, and Archive are not required.
Each Timeline represents a single contact and aggregates all communication with that person, including:

* Messages
* Live Chat
* Email

Items may also be filtered by communication type.

Within each Timeline:
* The newest message or email always appears at the top.
* The current draft always appears directly below the latest conversation.
* Drafts are automatically saved when leaving the page and continue editing when reopened.

## Gmail Attachment Flow

  User
    ↓
 Contact
    ↓
Gmail API
    ↓
Gmail stores the email and its attachments

After the email has been sent:
Gmail
└── Sent
     └── Document.pdf

The attachment already exists inside Gmail.
When the user later views the email in Contact:
Contact
    ↓
Gmail API
    ↓
Retrieve attachment metadata
    ↓
Display attachment to the user

The attachment source always remains Gmail.
Contact does not store the attachment itself—it simply retrieves and displays it through the Gmail API.
