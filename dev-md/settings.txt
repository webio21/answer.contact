# Live Mail - Mailbox Activation & Payment Flow Requirement

## Overview

Live Mail uses a *Setup First → Activate & Pay* model.

Users should be able to complete mailbox setup before payment.

The setup process includes:
- Selecting mailbox address
- Adding and verifying domain
- Creating mailbox configuration
- Setting password

After setup completion, the mailbox is saved as a *Pending Activation* state.

The actual mailbox must not be created through the reseller email API until payment is successfully completed.

This flow reduces signup friction, improves conversion rate, and avoids creating unpaid mailboxes that consume reseller resources.

---

# Mailbox Lifecycle

## Pending Activation

After the user completes mailbox setup:

- Save mailbox configuration data.
- Do not create mailbox through reseller API.
- Do not allocate mailbox storage.
- Do not start subscription billing.

Mailbox status:
Pending Activation

The saved data includes:

- User ID
- Domain
- Mailbox address
- Password hash
- Domain verification status
- Creation timestamp

The user can return later and continue activation.

---

## Activation

When the user clicks:
Activate Mailbox

The system starts the activation flow.

Activation flow:
Pending Activation
        ↓
Activate Mailbox
        ↓
Choose your plan
        ↓
Select Monthly / Yearly billing
        ↓
     Payment
        ↓
Payment Success
        ↓
Create Mailbox via Reseller API
        ↓
Mailbox Active
---

# Payment Flow

After the user selects a plan:

1. Create payment order.
2. Process payment.
3. Confirm payment success.

The mailbox must not be created before payment confirmation.

Correct flow:

      ↓
   Payment
      ↓
Payment Confirmed
      ↓
Create Mailbox

---

# Reseller API Integration

The reseller email API should only be called after successful payment.

Required flow:
Payment Success
        ↓
Call Reseller Mailbox API
        ↓
Create Mailbox
        ↓
Assign Storage Quota
        ↓
Enable Mailbox
        ↓
Update Status: Active

Before payment:
Pending Activation

After successful mailbox creation:
Active

---

# Payment Failure Handling

If payment fails:

- Do not call reseller API.
- Keep existing mailbox setup data.
- Keep mailbox status as:
Pending Activation

The user can retry activation later.

---

# Reseller API Failure Handling

If payment succeeds but mailbox creation fails:

System should:

- Keep payment record.
- Do not charge the user again.
- Retry mailbox creation.
- Show temporary activation status.

Example:
Activating Mailbox...

After successful creation:
Active

---

# Technical Requirements Summary

## Before Activation

The system stores:
Mailbox Setup Data

It is not an actual mailbox account.

No reseller mailbox resource is created.

---

## After Activation

The system creates:
Active Mailbox Account

through the reseller API.

The activation process separates:
1. User configuration
2. Payment confirmation
3. Mailbox provisioning

This prevents unpaid mailbox creation while providing a lower-friction signup experience.

## Live Chat Icon Customization 

- Free users cannot customize the chat icon and will use the default Starman Avatar.
- Pro users can customize the chat icon with Solid Color.
- Premium users can customize the chat icon with Image/GIF and Animated Video.

- Paid users can customize the chat icon using one of the following options:
  1. Starman Avatar
     - Use the default Starman Avatar as the chat icon.

  2. Solid Color
     - Use a solid color as the chat icon.
     - Store a single color value as Chat Icon Color.

  3. Image / GIF
     - Allow users to upload image assets, including JPG, PNG, and animated GIF files.
     - Provide image crop and scale adjustment to help users optimize the icon appearance.
     - Allow users to use their Card Avatar as the chat icon.

  4. Animated Video
     - Allow users to upload a welcome video asset.
     - Provide video crop and scale adjustment to help users optimize the animated icon appearance.
     - Automatically loop the video preview.
     - Display the video as an animated chat icon.
     - When the chat icon is opened, display the full uploaded video in the chat widget window.

* Additional configuration options are displayed only after the corresponding icon type is selected.

## Email Integration

Purpose:
Allow users to connect external email accounts to Contact Inbox.

Supported providers:

1. OAuth-based providers:
- Gmail
- Outlook

2. IMAP/SMTP-based providers:
- Domain Email
- Other email providers


## OAuth Email Providers

Gmail:
- Use Google OAuth authentication.
- Request required permissions:
  - Read emails
  - Send emails
  - Access email metadata

Outlook:
- Use Microsoft OAuth authentication.
- Request required permissions:
  - Read emails
  - Send emails
  - Access email metadata


## Domain Email Integration

Purpose:
Allow users to connect existing domain-based email accounts to Contact Inbox.

Must support:
- IMAP for receiving emails
- SMTP for sending emails

Connection flow:

1. User enters email address and password.
2. Detect provider settings automatically using:
   - Email domain
   - DNS MX records
   - Provider configuration database
3. Attempt IMAP/SMTP connection.
4. If successful:
   - Save email configuration.
   - Display connected email account in Integrations.
5. If automatic detection fails:
   - Show Manual Mail Server Setup UI.
   - Allow users to enter IMAP/SMTP settings manually.

Manual Mail Server Setup:

IMAP:
- host
- port
- encryption/security

SMTP:
- host
- port
- encryption/security

Password Handling:
The password field may accept:
- Normal email password
- Provider-specific app password

Do not expose "App Password" terminology by default.

If authentication fails and the provider requires an app password, display:
"Your email provider may require an App Password.
Please create one in your email provider settings and try again."


## Future Expansion (v2): First-party Email Service

The current MVP focuses on connecting existing email accounts through IMAP/SMTP.

The email integration layer should be designed to remain extensible, allowing future support for a first-party hosted email service via Email Reseller API.

Future capabilities may include:
- Managed email account provisioning
- Mailbox lifecycle management
- Domain-based email services
- Integration with Contact Inbox workflows

Do not tightly couple the current implementation to external email providers only.
Keep the architecture flexible for future hosted mailbox support.


## Calendar Integration

Purpose:
Allow users to connect an external calendar to Contact Meetings.

Supported providers:
- Google Calendar
- Outlook Calendar

Authentication:
- OAuth

Capabilities:
- Read calendar availability.
- Create and manage meetings.

Rules:
- Each Contact account can connect only one calendar provider.
- Only one calendar provider can be active at a time.
- Connecting a new provider replaces the current calendar connection.
- The previous calendar connection is removed after the new provider is successfully connected.

## Integrations

* Email Integrations
- Allow users to connect multiple email providers.
- Supported providers:
  - Gmail
  - Outlook
- Display all connected email accounts.
- Each connected account should show:
  - Provider name
  - Email address
  - Connection status
- Allow users to remove connected email accounts.
- Allow users to add additional email accounts.
- Require provider authentication and permission approval when connecting an account.
- Store connected accounts securely using provider authorization tokens.

* Booking Calendar Integrations
- Allow users to connect external calendars for availability management.
- Supported providers:
  - Google Calendar
  - Outlook Calendar
- Display the currently connected calendar account.
- Allow users to disconnect the connected calendar.
- Allow users to connect another calendar account.
- Use connected calendars to determine available meeting slots.
- Calendar availability should sync automatically after connection.
- Require provider authentication and permission approval when connecting a calendar.

* Email Sync
- Allow users to sync email messages from connected accounts.
- Imported emails should appear in the user's Inbox workspace.
- Preserve the original sender, recipient, timestamp, and thread information.

* Calendar Privacy
- Only use calendar availability data.
- Do not expose private calendar event details.
- Users control calendar connection visibility.
